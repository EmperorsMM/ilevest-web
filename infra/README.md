# infra/ — environments, deployment & the secrets policy

This is the CISO-facing operational doc. Two companion files:
- this file — **secrets policy** (what lives where, and how to get access)
- `environments.md` — the three environments, three Supabase projects, and deploy flow

---

## Golden rule
**No secrets in any repository, ever.** Not in code, not in config, not in history. The only
secret-shaped file that is tracked is `.env.example`, which contains variable **names and
placeholders only**. Real values live in three places, by context (below).

## The one distinction everyone must understand
Supabase issues two kinds of key, and confusing them is the single most dangerous mistake:

| Key | Plain-English meaning | Safe in browser/phone? | Where it may live |
|---|---|---|---|
| **anon / publishable** (`NEXT_PUBLIC_SUPABASE_ANON_KEY`) | A *public* key. Data is protected by Row-Level Security, **not** by hiding this key. | **Yes** | Client bundle, Vercel, GitHub Secrets |
| **service role / secret** (`SUPABASE_SERVICE_ROLE_KEY`) | A **master key that bypasses ALL Row-Level Security.** Anyone holding it can read or change anything. | **NEVER** | Server-side env + secret store only |

If the service-role key ever reaches a browser, a phone app, a log, or a repo, **the entire
trust model is broken** — it lets a human read or alter records the system is built to protect.
It must never be prefixed `NEXT_PUBLIC_`, never shipped to the Oracle app, and rotated
immediately on any suspicion of exposure.

## Where each kind of secret lives (by context)

1. **Local development → a developer's own `.env.local`**, populated from the **team secret
   manager** (recommended: 1Password / Doppler / Bitwarden — team's choice). `.env.local` is
   git-ignored. Developers get **dev** credentials only; never staging/production locally.

2. **CI (GitHub Actions) → GitHub Secrets, scoped by GitHub Environment.**
   - `staging` environment holds staging secrets.
   - `production` environment holds production secrets **and is gated** (required reviewers;
     only Admins can approve a production deploy). This means a human must approve before any
     production deployment runs — an intentional control, not friction.

3. **Runtime for the web app → Vercel Environment Variables**, scoped per environment
   (dev/preview, staging, production), each pointing at the matching Supabase project.

## Secret inventory (current + reserved)
| Secret | Sensitivity | Lives in | Issued by |
|---|---|---|---|
| `NEXT_PUBLIC_SUPABASE_URL` (per env) | Low (env-specific) | Vercel, GitHub Secrets | Supabase project settings |
| `NEXT_PUBLIC_SUPABASE_ANON_KEY` (per env) | Low (client-safe) | Vercel, GitHub Secrets | Supabase project settings |
| `SUPABASE_SERVICE_ROLE_KEY` (per env) | **CRITICAL** | Secret store + server env only | Supabase project settings |
| `SUPABASE_DB_URL` (per env) | **High** (DB superuser-ish) | GitHub Secrets (CI migrations) | Supabase DB settings |
| `VERCEL_TOKEN`, `VERCEL_ORG_ID`, `VERCEL_PROJECT_ID` | High | GitHub Secrets | Vercel account |
| `PAYSTACK_SECRET_KEY` *(reserved, later)* | **CRITICAL** | Secret store + server env only | Payment gateway |
| `PAYSTACK_WEBHOOK_SECRET` *(reserved, later)* | **CRITICAL** | Secret store + server env only | Payment gateway |
| Android signing keystore + passwords | **CRITICAL** | CI secret store / Play App Signing | Generated by Android lead |

## How a developer obtains access (least privilege)
1. Request access from an **Admin** (and, for production-tier, the **CISO**).
2. Admin adds them to: the GitHub repo/team (appropriate role), the Vercel team, and the
   secret-manager vault for the **lowest tier they need** (usually dev only).
3. No one gets production secrets by default. Production access is granted narrowly and is
   visible/auditable.

## Rotation
Rotate any secret on suspected exposure. Rotating the service-role or DB credentials
invalidates running server credentials — coordinate the swap (update the secret store +
GitHub Secrets + Vercel together).
